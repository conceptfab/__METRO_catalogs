## Executive summary — 3 najważniejsze rzeczy

1. **Cały łańcuch shadcn/ui został w praktyce wycięty z UI, ale żyje w runtimie i `package.json`.** `TooltipProvider` jest zamontowany w [src/app/providers.tsx](src/app/providers.tsx), ale **nie ma w kodzie ani jednego `<Tooltip>`/`<TooltipTrigger>`/`<TooltipContent>`**. Identycznie `<Sonner />` Toaster wisi w drzewie, ale `toast()` nie jest wywoływane nigdzie. To ciągnie 6 zbędnych paczek runtime: `@radix-ui/react-tooltip`, `sonner`, `next-themes`, `class-variance-authority`, `tailwind-merge`, `clsx`. Cleanup to ~5 plików + ~6 deps; bundle savings rzędu 30–60 KB JS.

2. **Drobne, ale realne błędy higieny w `package.json`.** `pnpm@^11.0.8` jest wpisany jako **runtime dependency**, `vercel@^53.2.0` (CLI deploy) też jest w runtime deps. Oba powinny być w devDeps lub całkowicie usunięte (`packageManager: "npm@10.9.2"` już jest ustawiony).

3. **`vercel.json` ma `deploymentEnabled: { stage_2: true }` i NIC poza tym.** To gating — po merge do `main` automatyczny deploy się **nie odpali**. Może być zamierzone (work-in-progress), warto potwierdzić.

## 4. Optymalizacje

### 4.1 Over-clientification w QX layout (HIGH impact, MEDIUM effort)

Wszystkie 10 sekcji QX (`HeroQX`, `OverviewQX`, `MaterialsQX`, `FeaturesQX`, `GalleryQX`, `FinishesQX`, `GettingStartedQX`, `ProductCodesQX`, `DimensionsQX`, `PackshotsQX`) ma `'use client'`. Powodem jest najczęściej `framer-motion`. **Wzorzec do rozważenia:**

```tsx
// HeroQX.tsx (Server Component)
export default function HeroQX(props) {
  return (
    <section>
      {/* statyczna treść, server-rendered */}
      <ClientMotionWrapper>
        {/* tylko fragment animowany */}
      </ClientMotionWrapper>
    </section>
  );
}
```

ROI: zmniejszenie hydratable JS, lepsze TTI. Wymaga audytu sekcja-po-sekcji — nie wszędzie da się rozdzielić bez kosztu w czytelności.

### 4.2 Skrypty image-pipeline — duplikacja

[scripts/process-images.mjs](scripts/process-images.mjs), [scripts/generate-thumbnails.mjs](scripts/generate-thumbnails.mjs), [scripts/recompress-gallery-bases.mjs](scripts/recompress-gallery-bases.mjs), [scripts/convert-png-to-webp.mjs](scripts/convert-png-to-webp.mjs) duplikują:
- `__dirname`/`__filename`/`ROOT`/`PUBLIC` boilerplate,
- ładowanie `sharp` z try/catch,
- helpery `fileExists`, `dirExists`,
- regex pomijania wariantów `-Nw.webp`.

**Sugestia:** [scripts/lib/image-utils.mjs](scripts/lib/image-utils.mjs) (paths, sharp loader, predykaty). Wzorzec już istnieje — [scripts/lib/section-widths.mjs](scripts/lib/section-widths.mjs).

### 4.3 [src/components/catalog/ColorChip.tsx](src/components/catalog/ColorChip.tsx) — `<img>` zamiast `next/image`

Linie 60-66 i 75-79 — `<img src={option.thumbnail}>`. Renderowane w 24px (chip) i ~116px (preview). Mała wygrana, ale konsekwencja w stosunku do reszty galerii. **Niski priorytet** — pliki swatch są małe, a chip ma `aria-hidden`.

### 4.4 Lato 5 wag — payload fontów

[src/app/layout.tsx:9](src/app/layout.tsx#L9): `weight: ['100', '300', '400', '700', '900']`. Sprawdzić, czy 100 i 900 faktycznie używane (np. `grep -rn "font-thin\|font-black"`). Jeśli nie — usunąć skrajne wagi, oszczędność po stronie klienta na każdym ładowaniu strony.

### 4.5 [tailwind.config.ts](tailwind.config.ts) — werbalność

Linie 20-92: każdy kolor zdefiniowany jako `'rgb(from var(--X) r g b / <alpha-value>)'`. ~40× powtórzony pattern. Akceptowalne (kompiluje się do poprawnego CSS), ale rozważyć helper TS:

```ts
const cssColor = (v: string) => `rgb(from var(--${v}) r g b / <alpha-value>)`;
// ...
'sidebar-accent': cssColor('sidebar-accent'),
```

Czytelność, brak różnicy w runtime. **Niski priorytet.**

### 4.6 Brak `decoding="async"` na `<img>` poza-fold

W kilku komponentach QX layout używane są `<img>` bez `decoding="async"` ani `loading="lazy"` (poza fold). Mała wygrana, ale `decoding="async"` jest tani do wprowadzenia.

### 4.7 [src/layouts/qx/PackshotsQX.tsx](src/layouts/qx/PackshotsQX.tsx) — z-index inline lightbox

Z pamięci sesji wynika znany bug: inline lightbox w `PackshotsQX` używa `z-[60]` (poniżej `z-modal=80`), podczas gdy globalna `Lightbox` używa już prawidłowego `z-modal`. Doprowadzić do spójności — `z-modal` wszędzie.
